Import a form

ProxForm reads two formats: a YAML-style indented file (human-friendly, easy to write in any text editor) or JSON (machine-friendly, what the Export button produces). Paste either into the box at the bottom of this page and click Create form.

The two-minute version

A complete form definition looks like this:

title: Pre-appointment intake
description: Please fill this in before your visit.

# Patient Information
text*    half    Full name
date*    half    Date of birth
text     half    Phone
text     half    Email

# Reason for Visit
textarea*        Primary reason for visit
yesno            Have you been treated for this before?

Each non-blank line is either form metadata, a section header, a field, or an option attached to a field above. The whole file is saved locally in your browser — never uploaded anywhere.

Syntax reference

Form metadata

Two keys, both optional but both highly recommended:

title: Pre-appointment intake
description: Please fill this in before your visit.
  • title: — The form's name. Shows up in the patient's view and in your forms list.
  • description: — A one-line note shown under the title. Useful for instructions.

Section headers

A line starting with # creates a section band that groups the fields below it. Sections have no inputs — they're labels. An optional indented > line directly under the section attaches a short description that shows beneath the title.

# Patient Information
                 > Required so we can match you with your record.
# Medical History
# Consent & Signatures

Fields

The general field syntax is:

<type>[*] [half|full] <label>
  • <type> — one of the eight supported types (see below).
  • * — optional. If present, the field is required.
  • half — optional. Pairs the field on a row with the next half-width field, so two questions sit side-by-side. Default is full-width.
  • <label> — the question text the patient sees.

The eight field types

text

Single-line text input.

text*    half    Full name
text             Allergies (none if blank)

textarea

Multi-line text input for longer answers.

textarea*        Describe your symptoms
textarea         Anything else we should know?

number

Numeric input (with the browser's number keyboard on mobile).

number*  half    Weight (kg)
number   half    Height (cm)

date

Date picker.

date*    half    Date of birth
date             Date of last vaccination

radio

Single-choice from a list. Options go on indented - value lines directly below.

radio*           Preferred contact method
                 - Email
                 - Phone
                 - SMS

checkbox

Multi-choice from a list. Same indented-options syntax as radio.

checkbox         Existing conditions (tick all that apply)
                 - Diabetes
                 - Hypertension
                 - Heart disease
                 - Asthma
                 - None of the above

yesno

Two-button Yes/No choice. No options needed — the type implies them.

yesno*           Are you currently pregnant?
yesno            Have you had this treatment before?

section

Not used as section directly — use # at the start of a line instead. This is the type stored internally when you write #.

Help text under a field

A line starting with > right under a field (indented) attaches a one-line hint shown under the question on both the preview and the patient's view. The same syntax attaches a description to a section if it sits under a # header instead.

text*    half    Phone
                 > Best number to reach you during the day

number*          Weight (kg)
                 > Round to the nearest whole number

Comments

Lines starting with // are ignored. Useful for notes or to temporarily disable a field while you're iterating.

// TODO: ask legal whether we need to collect insurance ID here
// text*    half    Insurance member ID

text*    half    Full name

What gets validated

  • Unknown field types raise an error on import — only the eight listed types are accepted.
  • Every radio and checkbox needs at least one option, otherwise the import fails.
  • Pairs of consecutive half-width fields land on the same row. A lone half followed by a full just takes the left column.

JSON format

If you prefer to generate forms programmatically or want a strict, machine-readable format, ProxForm also accepts JSON. The shape is:

{
  "title": "Pre-appointment intake",
  "description": "Please fill this in before your visit.",
  "fields": [
    { "type": "section", "label": "Patient Information" },
    { "type": "text",     "label": "Full name",      "required": true, "column": "half" },
    { "type": "date",     "label": "Date of birth",  "required": true, "column": "half" },
    { "type": "text",     "label": "Phone",                              "column": "half", "hint": "Best number during the day" },
    { "type": "text",     "label": "Email",                              "column": "half" },
    { "type": "textarea", "label": "Primary reason for visit",          "required": true },
    { "type": "yesno",    "label": "Have you been treated for this before?" },
    {
      "type": "radio",
      "label": "How would you describe the pain?",
      "required": true,
      "options": ["None", "Mild", "Moderate", "Severe"]
    }
  ]
}

An exported form (with the Export button on the forms list) drops in directly — you can use it as a backup, share a template with a colleague, or generate forms from a script.

Common mistakes

  • Option line not indented. Options for radio and checkbox must start with whitespace, then -, then the value. An un-indented - value is treated as an unknown field type.
  • Forgot the required star. * goes immediately after the type, with no space: text*, not text *.
  • Lone half. half only pairs with the next half-width field. If you put half on the last field of a section, it sits alone.
  • Unsupported types. File uploads, signatures, and rich text aren't in v1 — only the eight types listed above.
